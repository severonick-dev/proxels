## 0. О проекте

**Proxels** — сервис продажи доступа к собственной VPN-инфраструктуре на базе Xray
(VLESS / VMess / Reality). У владельца есть несколько VPS (сейчас 2 пустых в Германии,
будет больше). На каждом VPS поднят Xray. Пользователь покупает тариф, получает
subscription-ссылку и вставляет её в клиент (Nekobox / Hiddify / V2RayTun).

Домен: **proxels.ru**. Юр. лицо: ИП (есть, значит доступна ЮKassa).
Текущее состояние: ссылки выдаются вручную, клиентов >50.
Социальные сети: **только Telegram-канал** https://t.me/proxels (везде на сайте — в шапке,
в футере, в контактах — выводить именно эту ссылку как единственный официальный канал
коммуникации). Не добавлять плейсхолдеры для VK/Instagram/Twitter и пр.

**Цель проекта:** автоматизировать продажу и выдачу доступа + красивый SEO-оптимизированный сайт.

### Приоритет этапов (строго в этом порядке)

1. **Сайт: лендинг + личный кабинет + оплата (ЮKassa)** — ГЛАВНОЕ, делаем первым.
2. Автогенерация subscription-ссылок собственной системой (не сторонней панелью).
3. Health-check нод и failover (мёртвые сервера исключаются из подписки).
4. Гайды (страницы с пошаговыми инструкциями по приложениям).

### Среда разработки и деплой (важно!)

- **Разработка ведётся локально** на ПК владельца (Windows 10, Node 24, Docker есть, pnpm
  устанавливается отдельно). Все команды и скрипты должны корректно работать на Windows
  (через bash-совместимый shell), а также на Linux (Ubuntu) на сервере.
- **Хостинг кода — GitHub:** `https://github.com/severonick-dev/proxels.git`. С первого же
  коммита настраиваем remote `origin` на этот репозиторий. Все этапы пушим туда.
- **Боевой деплой — позже**, на VPS (Ubuntu). Структура и docker-compose должны быть
  готовы так, чтобы `git pull && docker compose up -d --build` поднимало всё на сервере
  без переписывания путей. Никаких абсолютных путей с диска `C:\` в коде/конфигах.
- Все ENV-переменные брать из `.env`; в `.env.example` — полный список с пояснениями.
- В `docs/DEPLOY.md` (Этап 13) описать перенос: клонирование с GitHub, заполнение `.env`,
  установка Docker, certbot, запуск.

## 1. Технологический стек (использовать именно это)

**Frontend**

- React 18 + TypeScript + Vite
- React Router v6
- TailwindCSS + shadcn/ui (компоненты)
- Zustand (стейт) или TanStack Query для серверного стейта
- i18next (локализация ru/en)
- next-themes-подобный механизм для тёмной/светлой темы (или собственный через class на <html>)
- Framer Motion для анимаций лендинга

**Backend**

- Node.js (LTS) + TypeScript
- NestJS (предпочтительно — модульность, DI, валидация из коробки) ИЛИ Express + структурирование вручную. Выбери NestJS.
- PostgreSQL + Prisma ORM
- Redis (сессии, очереди health-check, кэш статуса нод)
- BullMQ (фоновые задачи: проверка нод, продление/деактивация подписок)
- JWT (access + refresh), httpOnly cookie для refresh

**Инфраструктура**

- Docker + docker-compose (postgres, redis, backend, frontend-nginx)
- Nginx как reverse-proxy + раздача статики React
- Certbot / Let's Encrypt для TLS

**Платежи**

- ЮKassa (YooKassa) — официальный SDK или REST API. Вебхуки для подтверждения оплаты.

**Почта**

- Транзакционные письма (подтверждение регистрации, чек, реквизиты доступа).
  Использовать SMTP (любой провайдер) через nodemailer. Вынести в отдельный модуль.

## 3. Требования к дизайну (КРИТИЧНО — заказчик хочет «нереально красиво»)

Прочитай и следуй принципам качественного фронтенд-дизайна. Не делай generic-AI-вид.

- **Темы:** тёмная (по умолчанию) и светлая. Переключатель в шапке. Сохранять выбор в localStorage. Тёмная тема — глубокие графитовые/тёмно-синие тона, неоновый акцент (фиолетово-голубой градиент в духе «прокси/сеть»), стекло-морфизм аккуратно, без перебора.
- **Языки:** ru / en, переключатель в шапке, дефолт ru. Все строки — через i18next, никакого хардкода текста в компонентах.
- **Адаптивность:** mobile-first. На смартфонах — бургер-меню (off-canvas drawer с анимацией). Проверять брейкпоинты sm/md/lg/xl.
- **Лендинг должен содержать секции:**
  - Hero с анимированным фоном (частицы/сетка/градиент), заголовок, подзаголовок, CTA «Попробовать» и «Тарифы».
  - Блок преимуществ (скорость, стабильность, failover между серверами, простота).
  - Блок «Как это работает» (3–4 шага с иконками).
  - Тарифы (карточки с ценой; цена ~150 ₽, разные периоды: 1/3/6/12 мес).
  - Блок «Поддерживаемые приложения» (Nekobox, Hiddify, V2RayTun) с переходом на гайды.
  - FAQ (аккордеон).
  - Футер с ссылками на оферту, политику конфиденциальности, контакты, реквизиты ИП.
- **Микроанимации:** появление секций при скролле (Framer Motion), hover-эффекты на карточках, плавные переходы тем.
- **Иконки:** lucide-react.
- Шрифты: подключить пару (заголовочный grotesk + читаемый sans). Не использовать дефолтный системный без настройки.

## 4. Юридические требования РФ (КРИТИЧНО, не пропускать)

Сайт должен соответствовать законодательству РФ. Это требование уровня «без этого
не запускаемся». Реализовать всё перечисленное и заложить расширяемость:

- **152-ФЗ** «О персональных данных»:
  - Согласие на обработку ПДн при регистрации (чекбокс с ссылкой на политику).
  - Страница «Политика конфиденциальности» (редактируемая, версионируется в LegalDoc).
  - Хранить факт согласия: `User.consentPdnAt`, `User.consentPdnVersion` (версия политики
    на момент согласия) — это страховка на случай проверки/обновления документа.
  - Право на забвение: эндпоинт и кнопка «удалить аккаунт» в ЛК — реально удаляет ПДн
    и анонимизирует записи в платежах/логах (заменяет email/имя на `deleted-user-<id>`).
  - Реестр операторов ПДн: в `docs/LEGAL.md` оставить чек-лист по подаче уведомления
    в Роскомнадзор (это делает владелец, не код).
- **Локализация хранения ПДн на территории РФ** (152-ФЗ ст. 18 ч. 5): в `docs/DEPLOY.md`
  явно зафиксировать, что Postgres с ПДн пользователей разворачивается на сервере в РФ.
- **Публичная оферта**: отдельная страница, принятие при оплате (чекбокс с проверкой
  на бэке), хранить `Payment.offerAcceptedVersion`.
- **Уведомление об использовании cookie**: баннер с согласием, выбор «только необходимые / все».
  Без согласия не загружать аналитику.
- **Реквизиты ИП** в футере и на странице оферты (ИНН, ОГРНИП, ФИО, контакты).
- **Чек ЮKassa** (54-ФЗ): при оплате через ЮKassa передавать данные для
  фискализации (receipt с позицией, ценой, НДС, email покупателя). ЮKassa отправит чек.

  чтобы владелец мог поправить юр. тексты без правки кода. Тексты-заглушки оставить

> ВАЖНО: ты не юрист. Готовые тексты оферты/политики дай как шаблон с явной пометкой,
> что владелец обязан проверить их у юриста перед запуском.

## 4a. Приватность пользователей и «no-logs» политика (КРИТИЧНО)

> Это один из главных принципов сервиса. Архитектура должна гарантировать, что владелец
> (админ) физически не может посмотреть, какие сайты посещают клиенты. Это не «галочка»,
> а свойство системы — реализуется в коде и в конфигах нод.

Принципы:

- **Шифрование трафика клиентов:** уже обеспечивается VLESS/Reality (TLS 1.3, X25519).
  Не отключать, не понижать. Не добавлять никаких MITM-прокси на нодах.
- **На VPS-нодах с Xray:**
  - В конфиге Xray уровень логов — `none` (или `warning` без записи access-логов).
    Access-логи доступа к URL **запрещены**. Запретить writeback в файл `access.log`.
  - Не ставить никакие sniffer'ы / mitmproxy / DPI-логгеры.
  - Резолвер DNS на ноде — без логирования запросов (опц. DoH/DoT через апстрим без логов).
- **На стороне backend / админки:**
  - Админ видит только: количество клиентов, статус нод, агрегированный трафик подписки
    (байты вверх/вниз, без разбивки по доменам), финансы (платежи, выручка).
  - Админ НЕ видит и в принципе не должен иметь возможности увидеть: список посещённых
    доменов/URL пользователем, тело запросов, IP-адреса назначения.
  - В коде backend нет ни одного эндпоинта, который возвращал бы такие данные.
    Если кто-то в будущем попробует добавить — это считается багом безопасности.
  - В логах backend (pino) не логировать заголовки/URL, идущие через Xray. Backend
    их и не видит, но прямо запретить трогать API нод сверх AddUser/RemoveUser/Stats.
- **Статистика трафика для биллинга:** разрешено получать через Xray Stats API ТОЛЬКО
  суммарные байты по UUID клиента (uplink/downlink). Это нужно для отображения остатка
  трафика в ЛК. Никакого per-destination.
- **Логи backend:** не логировать subscription-токены целиком (только хвост), не логировать
  UUID клиентов целиком, не логировать IP клиентов в долгом хранении (агрегировать или
  обнулять через 24 часа — для rate-limiting достаточно).
- **Подписочная ссылка:** subToken должен быть криптостойким (>= 32 байта URL-safe random),
  ротация по запросу пользователя (кнопка «обновить подписку» в ЛК — старый токен инвалидируется).
- В футере сайта и на странице «Политика конфиденциальности» отдельно явно написать:
  «Сервис не ведёт журналов посещаемых ресурсов. Передаваемый через VPN трафик зашифрован».

Документ `docs/PRIVACY-ARCHITECTURE.md` (создать на Этапе 10) — описание, как именно
это обеспечено технически: какие логи отключены, какие API не реализованы, как
проверить.

## 4b. Безопасность и защита от ботов (КРИТИЧНО)

> Сервис будет под атаками — и автоматизированными (боты регистрируют аккаунты, парсят
> цены, брутфорсят логин), и адресными. Закладываем защиту с самого начала.

**Защита от ботов и автоматизированных атак:**

- **CAPTCHA**: на формах регистрации, входа, сброса пароля, обращения в поддержку.
  Предпочтительно Yandex SmartCaptcha (актуально для РФ, не блокируется). Альтернатива —
  hCaptcha. Капчу подключать через бэкенд-валидацию (никогда не доверять клиенту).
- **Rate-limiting (`@nestjs/throttler` + Redis-store):**
  - `/auth/register`, `/auth/login`, `/auth/forgot-password` — жёсткие лимиты на IP+email.
  - `/api/sub/:subToken` — лимит на токен и на IP, защита от перебора токенов.
  - Общий глобальный throttling на API.
- **Защита от перебора subToken**: токены длиной >= 32 байта random (см. §4a), любые
  попытки несуществующих токенов → 404 без подсказок + увеличение задержки/блокировка IP.
- **Honeypot-поля** в формах регистрации (скрытое поле, заполненное ботом → отказ).
- **Подтверждение email** обязательно перед оплатой/выдачей подписки.
- **Webhook ЮKassa** — проверка подписи/IP-источника (whitelist IP YooKassa).
- **WAF-уровень на nginx:**
  - Ограничения `limit_req_zone`, `limit_conn_zone`.
  - Базовый набор `bad bots` user-agent блок-листа.
  - Опционально (в docs) — подключение Cloudflare/StormWall как фронта.
- **DDoS-устойчивость:** nginx с `worker_connections`, fail2ban на сервере (в docs/DEPLOY.md).
- **Защита админки:**
  - Отдельный subpath `/admin`, недоступен по обычной авторизации (отдельная роль + ОБЯЗАТЕЛЬНО 2FA: TOTP).
  - IP-allowlist для админки (через ENV, опционально).
  - Аудит-лог всех админских действий (`AuditLog`).
- **Общие практики безопасности:**
  - Хеширование паролей — argon2id (предпочтительно).
  - Helmet, строгий CORS (только `proxels.ru`), CSP-заголовки.
  - HTTPS only, HSTS, secure+httpOnly+sameSite=lax/strict для всех cookie.
  - Все DTO — `class-validator`, отказы — без утечки внутренней структуры.
  - Секреты — только через ENV, никогда не коммитить.
  - Зависимости — регулярный `pnpm audit` в CI.
  - SQL-инъекции исключены за счёт Prisma (не использовать `$queryRawUnsafe`).
  - XSS: React по умолчанию защищает, но user-content (если будет) — через DOMPurify.
  - CSRF: refresh в httpOnly+sameSite=strict; для не-GET с cookie — double-submit token.

Документ `docs/SECURITY.md` (создать на Этапе 1, дополнять по ходу) — чек-лист всех
защит, чтобы перед боевым запуском пройти по нему.

## 4c. SEO (важно — лендинг должен ранжироваться)

Лендинг и публичные страницы — с упором на SEO под рунет:

- **Server-side rendering / pre-render лендинга:** Vite + React по умолчанию — SPA, что
  для SEO плохо. Решение: использовать `vite-plugin-ssg` или `react-router` SSR режим
  для пре-рендера статических HTML лендинга, гайдов, юр. страниц. ЛК остаётся SPA.
  (Если SSR усложняет — минимально: `vite-plugin-prerender` для главной + /guides + /pricing.)
- **Мета-теги** на каждой публичной странице: уникальный `<title>`, `<meta description>`,
  Open Graph (og:title, og:description, og:image, og:url), Twitter Cards. Через
  `react-helmet-async` или аналог.
- **Структура заголовков**: один H1 на страницу, иерархия H2/H3.
- **Семантическая разметка**: `<header>/<main>/<section>/<footer>/<nav>`.
- **Schema.org / JSON-LD** на лендинге: `Organization`, `Product`/`Service` (для тарифов),
  `FAQPage` (для FAQ-секции), `BreadcrumbList` (для гайдов).
- **sitemap.xml** генерируется автоматически (плагин Vite), обновляется при изменении
  гайдов/юр. документов.
- **robots.txt**: разрешить индексацию публичной части, запретить `/api/*`, `/admin/*`,
  `/lk/*`.
- **canonical** URL на каждой странице.
- **Изображения**: WebP/AVIF, `loading="lazy"`, `alt` обязателен.
- **Производительность (это тоже ранжирующий фактор):**
  - Lighthouse Performance / SEO / Best Practices / Accessibility >= 90 на лендинге.
  - Code-splitting, tree-shaking, минимизация CSS.
  - Кэширующие заголовки на статику.
- **Контент**: на лендинге — релевантные ключи (VPN, прокси, VLESS, Xray, Reality,
  Россия, разблокировка) органично в тексте, без переспама. Тексты — на ru, для en —
  отдельные URL `/en/...`, `hreflang` атрибуты.
- **Аналитика**: подключить Yandex.Metrika (после согласия на cookie) с целями
  (регистрация, оплата). Метрика — в env, выключаема для dev.
- **Канонический домен**: один (с/без www) — настроить редирект в nginx.
- **Verification**: оставить место под верификацию Yandex Webmaster / Google Search Console
  (мета-тег через env-переменные).

Документ `docs/SEO.md` (создать на Этапе 7) — чек-лист SEO с инструкцией для владельца:
куда вписать ключи, как заменить заглушки текстов.

## 6. Логика выдачи доступа (своя система, без сторонней панели)

Это заменяет Remnawave. Реализуем сами.

1. После успешной оплаты создаётся `Subscription` со статусом `active`, `subToken` (random).
2. Для подписки на каждой **живой** ноде создаётся `XrayClient` с собственным UUID.
   Добавление клиента в Xray на ноде — через **Xray gRPC API (HandlerService AddUser)**.
   Реализуй модуль `XrayNodeClient`, который коннектится к `xrayApiAddr` ноды и
   добавляет/удаляет пользователя в инбаунд по `inboundTag`. Предусмотри fallback:
   если gRPC API на ноде не настроен — перезапись `config.json` + reload (через SSH/agent).
   Начни с gRPC-варианта, опиши в docs, как включить API-inbound на нодах.
3. **Subscription endpoint**: `GET /api/sub/:subToken` отдаёт текст подписки
   (base64-список VLESS/VMess URI) — формат, который понимают Nekobox/Hiddify/V2RayTun.
   В выдачу включаются ТОЛЬКО ноды со статусом `online`. Это и есть failover на уровне
   подписки: упавшая нода не попадает в список, клиент использует живые.
4. При истечении/отмене подписки — удалить всех её `XrayClient` со всех нод.

> Subscription-ссылка пользователя: `https://proxels.ru/api/sub/<subToken>`
> (аналог текущей `remnawave.udonrtx.ru/api/sub/...`).

## 7. Health-check и failover (этап 3)

- BullMQ repeatable job каждые N секунд проверяет каждую ноду:
  TCP-connect на порт инбаунда + (опц.) реальный probe через Xray.

- Кэш статуса в Redis, чтобы `/api/sub` отвечал быстро.
- Эндпоинт `/api/health/nodes` для админки.

## 10. Админка

Полнофункциональная (не «минимум»). Подключается на Этапе 12 после здоровья
нод и подписок. Все эндпоинты под `/api/admin/*`, защищены
`JwtAccessGuard + RolesGuard + @Roles('admin')`, обязательная 2FA TOTP при логине
админа, опц. `ADMIN_IP_ALLOWLIST` (см. §4b), КАЖДОЕ действие в `AuditLog`.

### 10.1 Пользователи

- Список: email, role, emailVerified, createdAt, кол-во активных подписок,
  итого заплачено в рублях. Поиск, фильтры (active/expired/deleted).
- Карточка пользователя: профиль, его подписки, его платежи (+ статусы),
  AuditLog действий ПО нему (изменения админами).
- Force-actions (всё с подтверждением и audit):
  - revoke all sessions (`refreshTokens.updateMany revokedAt`).
  - force-verify email (для саппорта когда юзер потерял доступ к почте).
  - force-delete (анонимизация = тот же UsersService.anonymize что в DELETE /auth/me).
- Никаких полей вида «история посещённых сайтов» — их не существует (§4a).

### 10.2 Подписки

- Список с фильтрами по status, plan, «истекает в ближайшие N дней».
- Карточка: связанный пользователь, план, endAt, subToken (последние 6 символов),
  список XrayClient'ов по нодам, последние ротации (через AuditLog).
- Force-actions: cancel, extend на N дней (по жалобе/гудвилл),
  rotate subToken (как пользователь, но от имени админа — audit).

### 10.3 Платежи

- Список с фильтрами (status, amount, date range, есть/нет subscription).
- Карточка: связанные user+sub, raw YooKassa metadata (только для admin).
- Refund через YooKassa API (если технически возможно — позже).
- Никакого экспорта ПДн пачкой — только постранично, чтобы не слить базу.

### 10.4 Тарифы (Plan)

- CRUD уже реализован в Этапе 4 (`/api/admin/plans`). Здесь — только UI поверх.

### 10.5 Промокоды (Promo)

Полноценный CRUD + статистика — см. §10.7 ниже.

### 10.6 Ноды (Xray VPS)

- Список с last health-check, текущий статус (online/offline/degraded), вес.
- Add/update/disable. Reload config / re-sync XrayClients.
- Никогда не показывать access-логи (их нет, см. §4a).

### 10.7 Промокоды — модель и логика

**Зачем:** реферал-кампании, скидки к ивентам, B2B-договорённости.

**Модель (Prisma):**

- `PromoCode { code, discountKind: 'percent'|'fixedRub', discountValue,
validFrom?, validUntil?, maxUses?, usedCount, perUserLimit (default 1),
appliesToPlanIds[] (пусто = ко всем), isActive }`.
- `PromoRedemption { promoCodeId, userId, paymentId(unique), discountRub }` —
  доказательство применения, для аудита и анти-абуза.

**Логика применения** (в `PaymentsService.createForUser`):

1. Если в DTO есть `promoCode` — `PromosService.applyForPayment(code, userId, planId)`:
   - Найти активный, не просроченный, не выбравший maxUses.
   - Проверить `appliesToPlanIds` (если задано).
   - Проверить per-user-limit через COUNT(PromoRedemption WHERE userId).
   - Посчитать `discountRub` (округление вверх).
   - Сложить операцию в pending-структуру для финальной транзакции.
2. В Postgres-транзакции:
   - Создать Payment с amountRub = planPrice - discountRub (не ниже 1 ₽).
   - Создать PromoRedemption (paymentId pending → fill после создания Payment).
   - Инкрементировать `PromoCode.usedCount` атомарно.
3. AuditLog: `promo.redeem` с meta { promoId, userId, planId, discountRub }.

**API:**

- `POST /api/promos/validate` (JwtAccessGuard) — live-проверка из формы оплаты.
  Возвращает либо `{ ok: true, discountRub }`, либо ошибку без раскрытия деталей
  (т.е. «не подходит для этого тарифа», «лимит исчерпан» — это OK).
- `POST /api/payments/create` — расширить DTO опциональным `promoCode`.
- `GET/POST/PATCH/DELETE /api/admin/promos` — стандартный CRUD под admin.
- `GET /api/admin/promos/:id/redemptions` — детализация использования.

**Anti-abuse:**

- Rate-limit на `/api/promos/validate`: 30/60s на пользователя.
- При reuse-detection на refresh-токенах — отзывать также активные промо?
  Нет, перебор. Достаточно perUserLimit.

### 10.8 Юр. документы (LegalDoc)

- Список slug+version; текущая опубликованная отмечается.
- Markdown-редактор. Bump version + publish (старая версия живёт для аудита).

## 11. Безопасность и эксплуатация (краткая выжимка)

- Админка — 2FA (TOTP) + опц. IP-allowlist + аудит-лог.
- Никаких URL/destination-логов трафика клиентов нигде в системе (см. §4a).

## 13. Чего НЕ делать

- Не хардкодить тексты (всё через i18n) и секреты (всё через env).
- Не использовать localStorage/sessionStorage для критичных токенов (refresh — httpOnly cookie).
- Не выдавать готовую оферту/политику как юридически проверенные — только шаблон с пометкой.
- Не включать мёртвые ноды в подписку.
- Не делать generic-дизайн «как у всех AI» — заказчик явно требует выдающийся внешний вид.
- **Не логировать трафик клиентов** ни в каком виде (URL, домены, IP назначения) —
  ни в Xray, ни в backend, ни в nginx (для проксируемых через VPN потоков). См. §4a.
- **Не добавлять эндпоинты/админ-UI**, через которые можно было бы вытащить активность
  пользователя (посещённые сайты, тело запросов). Технически — невозможно увидеть.
- Не добавлять плейсхолдеры для VK/Insta/Twitter/YouTube — единственная соц-сеть это
  Telegram `https://t.me/proxels`.
- Не использовать `prisma.$queryRawUnsafe`, `eval`, `child_process` с пользовательским вводом.
- Не пушить в `main` без прохождения CI (lint + typecheck + сборка).
- Не использовать абсолютные windows-пути (`C:\...`) ни в одном файле проекта.
